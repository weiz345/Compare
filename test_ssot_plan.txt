Single source of truth — test file design
==========================================

Problem
-------
We need one test definition that both Python and C++ read directly, without
duplicating cases in either language and without Python re-serializing tests
into a different format for C++.


Recommendation
--------------
Use one file: script.cases

It contains only INPUTS (build/query operations). Neither language embeds
test data in source code. Both runners parse the same file, execute the
same sequence, and print the same canonical output. verify.py diffs the
two outputs.

Golden expected indices from test_plan.txt are documentation only for now.
They are not duplicated inside script.cases unless we add optional expect
lines later.


Why this file format
--------------------
Custom line-based script, not JSON:

  + Human-readable and easy to edit by hand
  + Trivial to parse in Python (split lines)
  + Trivial to parse in C++ (no JSON library)
  + Matches the build/query call sequence directly
  + Comments supported with #

JSON would be nicer for Python but adds dependency or boilerplate on the
C++ side. YAML same issue.


File location
-------------
  script.cases        <- the only place test inputs live

Both runners receive the path as an argument (default: script.cases).


Grammar
-------
Keywords: build, query

A script is a sequence of operations, top to bottom.
Blank lines and # comments are ignored.

BUILD
  build
  <x> <y> ...        one point per line, zero or more lines
  <x> <y> ...

  Empty build (zero point lines before next op) is valid.

QUERY
  query <x> <y> ... k <int>

  Coordinates are floats. k is an integer.
  Everything before the token "k" is the query point.


Example (Test 1 + Test 2 from test_plan.txt)
--------------------------------------------
# basic_knn
build
0 0
1 0
0 1
5 5
query 0.9 0.1 k 2

# empty_index
build
query 0 0 k 3


Who reads what
--------------
                    script.cases
                         |
            +------------+------------+
            |                         |
     script_loader.py          script_loader.hpp
            |                         |
      run_script.py              run_script.cpp
            |                         |
       (stdout)                   (stdout)
            |                         |
            +------------+------------+
                         |
                     verify.py
                   (diff outputs)


Each language owns a small parser (~40-60 lines) that implements the same
grammar. The parsers are not generated from a schema — they are simple
enough to keep in sync by hand.


Parsed structure (same concept in both languages)
-------------------------------------------------
After parsing, both sides hold a list of operations:

  BuildOp { points: [[float, ...], ...] }
  QueryOp { point: [float, ...], k: int }

  Operation = BuildOp | QueryOp

Execution model (identical in both):
  index = BruteForceNN()
  for op in operations:
    if BuildOp:  index.build(op.points)
    if QueryOp:  results = index.query(op.point, op.k)


Canonical output format (indices only)
--------------------------------------
Both runners print one block per query so verify.py can diff text output.

  build ok
  query k 2
  1
  0

Rules:
  - After each build: print "build ok"
  - After each query: print "query k <k>"
  - Then one index per line (no distances)
  - If k <= 0 or index is empty: print "query k <k>" then nothing else

Example for query returning [1, 0]:

  query k 2
  1
  0

Example for empty result:

  query k 3


Verification flow
-----------------
  1. python3 run_script.py script.cases   > /tmp/py.out
  2. ./build/run_script script.cases      > /tmp/cpp.out
  3. diff /tmp/py.out /tmp/cpp.out

verify.py wraps steps 1-3 and prints PASS/FAIL with a unified diff.

We compare Python vs C++, not against embedded golden values. That proves
functional equivalence. test_plan.txt expected indices are the spec we
used to author script.cases and to eyeball failures.


Alternative considered: embed expected indices in script.cases
--------------------------------------------------------------
  query 0.9 0.1 k 2
  expect 1 0

Pros: one file holds inputs and expected outputs; verify could check each
      implementation against golden without cross-diff.
Cons: duplicates what both implementations should compute; more grammar;
      two ways to fail (wrong vs golden, mismatch vs each other).

Not recommended for v1. Cross-diff is simpler and directly answers
"are Python and C++ identical?"


Alternative considered: JSON single file
----------------------------------------
  script.json with [{ "op": "build", "points": [...] }, ...]

Pros: standard format, extensible.
Cons: C++ needs a JSON parser or Python must feed C++ (breaking SSOT on
      disk for C++).

Not recommended unless we add a header-only JSON library.


Alternative considered: Python-only loader pipes to C++
-----------------------------------------------------
Python reads script.cases, serializes to stdin for C++.

Cons: C++ no longer reads the source file; Python owns parsing logic.
Rejected — we want both to read script.cases directly.
